核心理念：从脚本到策略

Harness 的底层哲学是用"策略声明"替代"过程脚本" — 你声明目标状态和约束条件，平台自动执行并收敛。

🎯

声明式交付

YAML 定义部署目标、回滚策略、健康检查阈值。平台负责执行细节，而非你编写复杂的 Shell 步骤。

🧠

AI 驱动决策

内置机器学习引擎分析部署指标（错误率、延迟、吞吐），自动判断金丝雀/蓝绿推进还是自动回滚。

🔐

Secret 与 RBAC 原生集成

凭证管理内置在平台层，不随 Pipeline YAML 传递。细粒度 RBAC 控制谁能在什么环境执行什么操作。

# Harness 声明式 Pipeline YAML 示例
pipeline:
  name: microservice-deploy
  identifier: ms_deploy
  stages:
    - stage:
        name: Deploy to K8s
        type: Deploy
        spec:
          deploymentType: Rolling         # 策略声明，非脚本
          failureStrategy:
            onFailure:
              action: Rollback               # 自动回滚
          healthCheck:
            type: Datadog
            metric: error_rate
            threshold: < 1%

平台架构

从开发者体验到策略引擎再到资源调度 — 五层分层架构，每层独立可插拔。

CI/CD 核心流水线

Harness 的 CI 和 CD 是独立但深度集成的模块，共用 Delegate、Secret Store 和 RBAC。

Pipeline Stage 类型

Stage 类型作用典型触发关键能力
CI (Build) 代码检查、编译、单元测试、镜像构建与推送 Git Push / PR / Webhook 测试智能拆分、缓存层、多架构构建
CD (Deploy) 将构建产物部署到目标环境 CI 完成 / 手动审批 / 定时 金丝雀/蓝绿/Rolling、自动回滚
Approval 人工审批门控 部署前 / 金丝雀推进前 多级审批、Jira/SNOW 集成、邮件通知
Custom 自定义步骤（脚本、通知、Webhook 回调） 任意位置插入 Shell / Python / HTTP / Plugin 扩展

三种部署策略对比

🔀

Rolling（滚动）

逐个替换 Pod/实例，适合有状态服务。可设置最大不可用数和最大激增数。

🐛

Canary（金丝雀）

先部署 5% 流量到新版本，AI 自动分析指标，通过后逐步扩到 100%。

🔵

Blue-Green（蓝绿）

两套完整环境并行运行，流量一键切换。秒级回滚，但需要双倍资源。

流水线执行模型

Harness Pipeline 是 DAG（有向无环图）而非线性序列 — 支持并行 Stage、条件分支、矩阵执行。

🔄

Trigger → Pipeline → Stages → Steps

四层嵌套模型。Trigger 定义触发条件（Cron / Webhook / Artifact 变更），Pipeline 编排 Stage DAG，Stage 内 Steps 顺序执行。

📦

Artifact / Manifest 抽象

不直接写镜像名或 Helm Chart 路径。通过 Artifact Source 和 Manifest Source 抽象绑定，切换版本只需改 Trigger 条件。

♻

Looping / Matrix 策略

Matrix 语法在一个 Stage 内对不同环境/集群并行执行相同步骤。Looping 支持对列表元素迭代执行。

# Matrix 执行策略 — 同一 Stage 在 3 个环境并行部署
stage:
  name: Deploy Matrix
  strategy:
    matrix:
      env: [dev, staging, prod-us]
      region: [us-east-1, eu-west-1]
      exclude:
        - env: dev
          region: eu-west-1        # dev 仅部署 us-east-1

安全治理与策略即代码

OPA (Open Policy Agent) 作为策略引擎，Pipeline 执行前强制执行合规规则。

01

Pipeline 触发器激活

Git Push / PR Merge / Manual Run / Cron Schedule 触发 Pipeline 执行请求。

02

OPA 策略评估（Pre-Flight）

Harness Manager 调用 OPA 引擎，评估所有绑定的 Policy Set。例如："prod 环境禁止周末部署"、"必须关联 Jira Ticket"。

03

Delegate 领取任务

策略通过后，目标环境的 Delegate 通过出站 HTTPS 轮询从 Manager 领取 Stage 任务，无需开放入站端口。

04

执行 + 实时回传

Delegate 在集群/VPC 内执行 kubectl/terraform/helm 命令，日志和指标实时回传 Manager。

05

Governance 审计日志

每次执行完整记录：谁触发、策略评估结果、Delegate 行为、部署前后差异。满足 SOC2 / FedRAMP 审计要求。

Delegate 安全优势

出站轮询模式意味着无需在防火墙上开放入站端口。Delegate 运行在 VPC 内，拥有最小所需权限的 IAM Role。

Secret 管理

所有凭证存储在 Harness Secret Manager 或外部 Vault/AWS KMS/GCP Secret Manager。Pipeline YAML 中仅引用 Secret ID。

最佳实践

用好 Harness 的关键不是掌握 YAML 语法，而是理解其策略驱动模型。

✔ 推荐做法

  • Template 化 — Pipeline/Stage/Step 模板，通过 Git 同步复用
  • Environment + Infrastructure Definition 分离 — 环境定义与基础设施实现解耦
  • AI 验证金丝雀 — 生产部署必须绑定 Harness 自动验证或 Datadog/Prometheus 健康检查
  • 多 Delegate — 按环境/集群部署独立 Delegate，避免单点
  • OPA 门控 — 对所有生产环境 Pipeline 绑定合规策略
  • Feature Flags 解耦部署与发布 — 部署代码，用 Flag 控制上线
  • Pipeline as Code — YAML 存 Git，PR 审核变更

✘ 反模式

  • Pipeline 中硬编码凭证/密钥 — 始终用 Secret Manager 引用
  • 一个 Delegate 跑所有环境 — 生产隔离失效
  • 跳过 Canary 分析直接全量 — 失去 AI 自动回滚保护
  • Pipeline 变成万行 Monolith — 模块化拆分
  • 忽略 Delegate 版本升级 — 安全补丁和功能更新延迟
  • 用 Custom Stage 替代 CD Stage — 丧失声明式回滚和策略能力
  • 所有团队共用同一 Connector — 权限粒度不足
