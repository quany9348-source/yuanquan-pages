全局ID唯一

虽然分为三个独立品牌空间，但 WTI ID 在全局范围保持唯一性，从根源上杜绝跨品牌、跨车型的 ID 冲突。

纵向代际统一

单一品牌内不区分NT3.0和NT3.5平台，一份配置表覆盖该品牌下所有车型和代际的WTI条目。

Section 03

平台演进带来的核心矛盾

NT3.0 → NT3.5 底层电子电气架构演进，服务接口兼容性断裂

NT3.0 · 统一时代

多品牌所有系统的服务接口相互兼容

只有一版软件基线，最大化覆盖所有 WTI 需求

统一软件基线 → 全平台兼容
NT3.5 · 独立分支

服务接口发生重大变化，成为独立分支

实现自身多车型兼容，但与 NT3.0 互不兼容

独立软件基线 → 仅限 3.5 平台
核心矛盾：配置表旧模式 "3.0 与 3.5 合一" ↔ 服务接口 "3.0 与 3.5 割裂"
NT3.0 开发方式

VAS WTI 应用软件基于 Simulink 模型 进行开发。 每个 WTI 条目的报警逻辑以图形化模块搭建在 Simulink 模型中，通过模型自动生成 C 代码后集成到应用软件。 当配置表新增条目时，开发人员需手动在 Simulink 模型中逐一增加对应逻辑模块——每一次增量对应一次人工操作。

Section 03.5

Simulink 模型的负荷与瓶颈

在"双基线软件分叉"的新架构下，基于 Simulink 模型的开发模式面临五个核心挑战

01

双线维护成本倍增

跨平台共性的 WTI 条目需在两套 Simulink 模型中分别搭建相同逻辑，工作量翻倍

02

配置表增量的人工瓶颈

每个新增 WTI 需手动拖拽模块、连线、配置参数，效率极低且难以自动化

03

协作与版本管理缺陷

.slx 二进制/XML 混合格式难以 diff、merge、审查，并行开发冲突频发

04

测试与回归周期长

每次变更需重新生成代码、编译，执行 MIL/SIL/HIL 全链路测试，验证成本极高

05

跨基线逻辑一致性难以保证

同一 summary 逻辑在两套基线的人工实现可能存在差异，埋下质量隐患

继续沿用 Simulink 人工开发模式，将面临
人力投入线性膨胀 · 交付周期持续拉长 · 质量风险累积
的不可持续局面

必然路径： 手写代码（Hand-Code）替代 Simulink 建模 + AI 赋能

WTI 逻辑自动生成 + 双基线代码自动同步 + 配置表到代码自动映射
Section 04

最终解决方案：软件分线与配置表房间拆分

ROOM

数据层面：各品牌拆分 3.0 与 3.5 独立房间 Configuration Room Split

各品牌（NIO、ONVO、Mollary）在现有 NT3.0 配置表基础上，各新增一个独立的 NT3.5 配置房间。 需求方直接提交至对应平台房间，无需额外打标。

消除人工打标

需求方直接提交至对应平台房间，无需逐条评估兼容性

02

基线天然对齐

数据源与代码基线一一对应，逻辑清晰无歧义

03

独立演进

两平台配置表独立迭代，互不干扰，稳定性隔离

管理闭环

VAS 开发团队以配置房间的平台归属为唯一依据， 在两条独立的软件基线中分别提取、开发和测试对应的 WTI 功能。 两条基线各自对接专属的配置房间，从根本上消除了配置统一与接口断代之间的架构冲突。
