核心架构：Agent Loop

Agent 的底层运行逻辑是一个隐式状态机，分为 QueryEngine（会话管理）和 queryLoop（单轮执行）两层。

// Agent 执行循环的简化模型
while (claude_response.has_tool_call):
    result = execute_tool(tool_call)       // 执行工具
    claude_response = send_to_claude(result)  // 结果喂回模型
return claude_response.text                 // 最终回复
⚙

QueryEngine

会话管理 — 多轮状态追踪、transcript JSONL 持久化、SDK 协议适配、用量累加统计。是整个会话的"大脑"。

↻

queryLoop

单轮执行 — API 调用、工具执行、错误恢复。内含 7 种恢复路径和 10 种终止条件，保证鲁棒性。

↔

AsyncGenerator 桥接

两层通过 AsyncGenerator 连接：queryLoop yield 消息，QueryEngine 消费并转发。提供背压、级联取消和流组合能力。

Agent 分层体系（三层拓扑）

从轻量级子代理到团队协作再到纯编排 — 逐层递进的能力与抽象。

第一层：Subagent 详解

最常用的 Agent 类型，上下文隔离 + 工具过滤 + 可选隔离，是任务委派的基础单元。

类型能力范围适用场景典型耗时
Explore 只读搜索（Glob, Grep, Read），Haiku 模型 快速定位代码、关键词搜索、文件发现 5-15s
Plan 只读 + 架构分析 + 方案设计 设计多方案对比、评估技术路线、输出实施计划 15-45s
General-purpose 完整工具集（读写执行） 复杂多步骤任务、代码修改、调试、测试 30s-5min
claude-code-guide 读 + 搜索 + WebFetch 回答 Claude Code CLI / SDK / API 用法问题 10-30s

关键机制

上下文隔离

每个 Subagent 拥有独立的上下文窗口，不继承父会话历史，避免污染主会话的 token 预算。结束后只返回摘要。

Depth=1 限制

Subagent 不能再创建 Subagent，防止递归爆炸。这一硬限制确保系统可控。

工具过滤

可按白名单 / 黑名单控制 Agent 可用工具。例如 Explore 天然无 Write/Edit/Bash 权限，确保只读安全。

Worktree 隔离

创建临时 git worktree，Agent 所有文件操作在隔离目录进行。干净则自动清理，有变更则保留供审查。

第二层：Agent Teams / Swarm（实验性）

多个 Agent 组成团队，通过文件系统 mailbox 进行结构化消息传递，共享任务队列。

👥

Leader + Teammates

Leader 分配任务，Teammates 执行并回报。双向通信通过文件系统 mailbox 实现。

📨

Mailbox 消息协议

结构化协议消息：shutdown_request、plan_approval_request 等。

⏳

Idle 生命周期

Teammates 完成任务后进入 Idle 状态，等待新任务分配，避免反复创建销毁开销。

⚠

数量限制

TEAMMATE_MESSAGES_UI_CAP = 50。真实案例：2 分钟产生 292 个 agent → 36.8GB 内存，故设硬限制。

~/.claude/teams/
├─ <teamName>/
│ └─ mailbox/
│ └─ <agentName>/ # 每个 teammate 的 mailbox

第三层：Coordinator 模式

纯编排角色 — 不直接操作文件，只做任务拆解、委派和结果汇总。

01

Research（研究）

派 Explore / General-purpose Subagent 搜索代码库，收集相关信息。多个 Agent 可并行执行。

02

Synthesis（综合）

这一步由你来完成，不能委派！理解收集到的信息，形成判断，写出精确的实施指令。

03

Implementation（实施）

派 Subagent 执行具体修改。多个独立文件修改可并行进行。

04

Verification（验证）

派独立的 Subagent（无历史上下文）以"新视角"审查变更结果。

核心原则："Never delegate understanding"

编排者必须自己消化信息再写精确指令。把"理解"推给 Subagent 是最常见的反模式。

最佳实践

何时用 Agent、何时不用 — 直接影响任务效率和上下文预算。

✔ 推荐使用 Agent

  • 探索 10+ 文件 — 用 Explore 只读搜索，不占主会话上下文
  • 3+ 个独立文件修改 — 并行 Subagent 各自修改
  • 需要"新视角"审查 — 无历史上下文的 Subagent 做 unbiased review
  • 流水线工作流 — 链式 agent：设计 → 实现 → 测试
  • 长时间搜索/分析 — 后台运行，不阻塞主会话

✘ 不推荐使用 Agent

  • 小于 10 个文件的搜索 — 直接用 Glob/Grep 更高效
  • 强顺序依赖的任务 — 主会话串行执行更好
  • 需要完整会话上下文的任务 — Subagent 上下文隔离是劣势
  • 单个文件的简单修改 — 开销大于收益
  • 纯对话/问答 — 无需委派

效果指标

⚡

50-70% 耗时缩减

并行 Agent 处理多模块独立任务时，总耗时接近最长单个任务而非累加。

💾

主会话 Token 节省

Explore Agent 搜索不消费主会话上下文，保留预算给核心理解和决策。

🛡

零文件冲突

Worktree 隔离确保并行 Agent 的文件操作完全独立，无竞态条件。

文件体系

Agent 的定义、运行产物和持久化数据在文件系统中的完整布局。

└─ security-auditor.md

└─ my-explorer.md

└─ <agent-slug>/ 每个 Agent 独立的 git worktree
# Team 文件（实验性）
~/.claude/teams/
└─ <teamName>/
└─ mailbox/
└─ <agentName>/ 每个 teammate 的 mailbox
# Transcript 记录
~/.claude/transcripts/ Agent 执行过程 JSONL 日志
└─ <session-id>.jsonl
# 持久化 Memory
.claude/projects/<project>/memory/
├─ MEMORY.md Memory 索引
├─ user_*.md 用户记忆
├─ feedback_*.md 反馈记忆
├─ project_*.md 项目记忆
└─ reference_*.md 引用记忆
