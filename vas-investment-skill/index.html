简介

VAS（Vehicle Application Software）投入测算 Skill 面向 车辆应用软件部门， 在需求评审阶段对新提出的软件需求进行 系统化、可复现、可审计的工程人月与费用估算。

定位与价值

01 / AUDITABLE

财务审计级估算

每一条结论都绑定工程证据——架构数据字段、分级表触发词、历史案例数据。杜绝"拍脑袋"报价，做到逐项可追溯。

02 / DUAL-PATH

双路核算三角验证

自顶向下（历史统计回归）+ 自底向上（BU 任务目录展开），两路独立计算后对账。偏差 <20% 才给 High 置信度。

03 / RISK-ADJUSTED

风险量化建模

不报单点值：输出 P50（基线）/ P80（推荐）/ Max（上限）三档，将工程不确定性和历史经验纳入风险溢价。

04 / FEISHU-NATIVE

飞书原生集成

数据源全部来自飞书多维表格 / Wiki，分析结果可一键输出为飞书文档，含系统拓扑图、变更矩阵、资源总览。

适用场景

新需求投入评估 VAVE 降本测算 新车型平台适配 新架构迁移估算 多 BU 联合报价 预算评审支撑 历史项目复盘

用户关心的

面向三类角色：需求提出方（"做这个功能要多少钱？"）、 BU / EM（"我的团队要投多少人月？"）、 财务 / 管理层（"这个数字能不能审计？"）。

能做什么

需求 → 工程定级 → 人月估算，全自动

用户只需用自然语言描述需求（"在 L60 上增加方向盘加热功能"），Skill 自动完成功能定位、IO 信号映射、SOA 服务分析、L1-L6 定级仲裁、历史案例检索、双路人月计算，最终输出一份完整报告。

多 BU 协同场景，逐 BU 拆分明细

一个需求往往涉及多个 BU（如 Body + HVAC + VMBC 平台），Skill 会逐 BU 展开任务目录，列出每个 BU 的具体工作项和对应人月，让每个 BU 的 EM 可以审核自己那部分。

不止人月，还有完整费用结构

报告涵盖 PG 费用、环境仓费用、样件费用、车辆使用周期等多维度费用结构，适配完整的财务审批流。

一键输出飞书文档

分析完成后，报告可直接发布为飞书文档，包含系统拓扑图（Mermaid）、变更矩阵、定级仲裁表、双路对账表、资源总览等，方便团队协作和审批流转。

数据可同步更新

历史报价、BU 任务目录、工程分级表均支持从飞书一键同步刷新。说"同步数据"或"更新 catalog 参数"，Skill 自动拉取最新数据。

依赖检查 + 分级阻塞反馈

分析启动前自动检查所有依赖项（数据文件、认证状态、架构索引），分级反馈：阻塞（必须解决）vs 降级（可继续但置信度降低）。

Skill 产出完整报告：执行摘要 → 技术方案 → 工程定级 → 预算评估（双路对账 + 三档模型）→ 开发计划。可选择输出到飞书文档。

原理和逻辑

为什么能做到"经得起财务审计"？核心在于将模糊的经验判断拆解为 可机械执行的工程仲裁规则，通过 历史数据统计 → 双路独立核算 → 交叉验证三层防御体系，杜绝单点偏差。

SKILL架构

计算数据流

核心机制 1：L1-L6 六维独立判级 + MAX 仲裁

传统做法是"新增 IO → L4"这种单一触发词锁定全局等级，会丢失不同工程维度的精度。 本 Skill 将工程变更拆解为 6 个独立维度，每维独立判级，最终取最高值—— 宁可高估不可低估。

维度名称关注内容仲裁规则
D1System Solution系统方案、跨系统接口、拓扑变化R1 信号增量
D2Hardware Solution芯片、传感器、执行器、控制器硬件变化R2 服务增量
D3Software Solution4 子维：sw_architecture / service_signal_interface / sw_unit_logic / sw_unit_deploymentR3 部署增量
D4Industrialization量产、供应链、发布影响R4 ZONE 拓扑
D5Integration & Virtual Validationlabcar、仿真、集成验证复杂度R5 HW 增量
D6Physical Validation实车、台架、道路或法规验证复杂度R6 BU 归属
L-GRADE D3 = MAX( sw_architecture, service_signal_interface, sw_unit_logic, sw_unit_deployment ) L_final = MAX( D1, D2, D3, D4, D5, D6 ) D1 系统方案 / D2 硬件方案 / D3 软件方案 / D4 工业化 / D5 集成验证 / D6 物理验证

※ AI 语义初判与机械判级冲突时，以机械判级为准。无法机械判定的维度需显式标记依赖人工确认。

核心机制 2：变更方向系数 (direction_factor)

L 等级反映工程复杂度，但同一 L 等级下"新增一个功能"和"修改一个已有功能"的实际投入可差 3-10 倍。direction_factor 作为 L → 人月映射的第二维乘数，不改变 L 定级本身，在任务级应用方向系数。

方向默认系数典型工作量构成架构类任务测试类任务标定任务
new1.0从零设计 → 开发 → 单测 → 集成 → 验证 → 文档1.01.01.0
modify0.6影响分析 → 代码修改 → 回归测试 → 文档更新0.50.70.6
delete0.3依赖追踪 → 代码/配置清理 → 回归验证0.2—0.0
DIRECTION PM_j = Σ_t [ ( B_{j,t}(L) + n_t · c_{j,t}(L) ) × d_{j,t}(dir) ] d_{j,t}(dir) ∈ { new: 1.0, modify: 0.6, delete: 0.3 } B=固定基准工时 / n=缩放变量计数 / c=变量单价 / d=方向系数 / dir=变更方向

核心机制 3：BU 任务目录结构

两套模型覆盖全部 11 个 BU，每项任务按 L1-L6 预置基准人月系数和 direction_factor 方向乘数。

模型 A — VMBC 平台专属（21 项任务）

架构设计（11 项）：平台化 IO 系统设计 / 新芯片方案及评估 / Zone·CCC 最小系统失效分析 / CPM 电源及备份供电 / 功能架构设计 / 时延分析与端到端集成验证 / E2E 性能 (Body·Motion·Light) / 资源管理方案 / 平台服务接口设计与维护 / TOX2 协议升级适配 / S2S mapping 整体管理

复杂驱动开发（1 项）：新芯片复杂驱动开发与 B0 样件适配

业务中间件开发（8 项）：通讯及设备封装类中间件 / 上下电·网络管理 / ACMS 数据上传 / 诊断管理系统方案 / 计算资源管理 / VDF-A 核心部署中间件 / TOX2 协议兼容性开发 / S2S 映射运行时维护

横向支援（1 项）：配合功能 BU 完成平台适配

模型 B — 功能 BU 通用模板（7 类任务，覆盖 Motion X/Y/Z、VMC、Body、HVAC、HV/LV、Seat、Lighting、AI 技术平台 共 10 个 BU）

软件应用层开发（scalable_by_count）— 按功能点数 × 信号数线性缩放，ASIL 等级加权

单元测试（scalable_by_count）— 与开发量正相关，ASIL-D 测试工作量倍增

集成测试和问题闭环（fixed_by_L）— L 等级决定测试深度，跨 BU 场景增加集成复杂度

系统架构设计（fixed_by_L）— 跟 D1 维度，L 越高架构设计量越大

软件架构（fixed_by_L）— 跟 D3.sw_arch 子维

项目管理（fixed_by_L）— L 等级基准 + 可选按比例（其他任务之和 × 15%）

系统标定（scalable_by_count）— AI 技术平台除外，delete 方向不触发

核心机制 4：双路核算 + 交叉验证

两条完全独立的计算路径分别产出估算，然后对账——这是系统最终的检验关口。偏差过大说明存在未知风险或参数错误，需要强制复盘而非强行输出一个数字。

路径 A  ·  自顶向下  —  Step 5

基于 104 条历史评估记录，筛选同 L 等级 + 同变更类型 + 同架构代际的相似项目，取 P50 / P80 / Max 三档作为估算区间。

TOP-DOWN P50 = PERCENTILE(同类样本, 0.50) P80 = PERCENTILE(同类样本, 0.80) Max = MAX(同类样本) P50=中位数基线 / P80=80分位推荐值 / Max=历史最高上限
继承真实项目隐性风险，反映"实际花了多少"
L5 / L6 样本可能不足（n < 3），此时自动降级，以自底向上为主路径

路径 B  ·  自底向上  —  Step 7

将需求拆解为变更矩阵（每条信号 / 服务变更的明细），逐 BU 匹配 catalog 中的任务模板，按 L 等级 + 数量变量 + 方向系数逐项累加计算。

BOTTOM-UP PM_j = Σ_t [(B(L) + n·c(L)) × d_factor] PM_bot = (Σ_j PM_j) × (1 + αR4 + αR6) PM_j=单BU人月 / PM_bot=自底向上总计 / αR4=拓扑溢价 / αR6=跨BU联调溢价
粒度细，可逐任务审计，BU / EM 可以直接审核自己那部分
初期 catalog 系数需经 BU 校准和历史回归验证

防御性设计原则

PRINCIPLE 1

保守定级

六维取 MAX 而非平均或加权，L 等级冲突时取高不取低，确保不因某维度定级过低而低估整体投入。

PRINCIPLE 2

全证据链

每条结论必须引用数据来源（JSON 字段路径 / 分级表触发词 / 需求原文段落）。无法判定的维度显式标记。

PRINCIPLE 3

报告可审计

最终报告包含六维仲裁表、方向判定、双路对账过程、历史案例参考、逐 BU 任务明细，完整复现从需求到结论的全链路。

PRINCIPLE 4

小样本退化

L5 / L6 级历史样本不足 3 个时，自底向上自动升为主路径，自顶向下仅作参考上限，防止小样本统计偏差。

PRINCIPLE 5

大偏差强制复盘

Δ ≥ 50% 不输出结论，强制回归检查：是 catalog 系数问题、历史样本异常值、还是需求本身多变性导致？

PRINCIPLE 6

数据分区隔离

架构数据（只读）与投入测算数据（可写）严格分区，架构数据更新必须通过 nt35-data-cache skill，防止交叉污染。
